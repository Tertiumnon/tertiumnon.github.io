# ИИ: реальность после хайпа

## Преамбула

Я начал писать код с помощью ИИ примерно в мае 2025 года. За год непрерывной работы с Claude Code и другими ИИ-помощниками я накопил много наблюдений, ошибок и способов, которые действительно работают.

Эта статья — не общие рекомендации из интернета. Это то, что я выучил на собственной шкуре: какие подходы сбережут вам деньги и нервы, какие инструменты реально полезны, и самое важное — как организовать процесс разработки, чтобы ИИ был помощником, а не источником проблем.

Структура простая: сначала мы разберёмся в реальности (что оправдалось, что нет), потом я дам вам систему, которая работает.

# Часть I: Реальность ИИ в 2026

## 1. Обещания vs Реальность

### Что обещали

В 2023-2024 ИИ-компании обещали:
- ИИ заменит большинство белых воротничков
- Произойдёт эпоха изобилия
- Будут научные прорывы
- Коды будут писаться полностью автоматически

### Что на самом деле произошло

[MIT Technology Review отмечает](https://www.technologyreview.com/2025/12/15/1129174/the-great-ai-hype-correction-of-2025/), что 2025 стал годом "великой коррекции хайпа вокруг ИИ". Обещания не исполнились.

[По данным Stanford AI Index](https://hai.stanford.edu/news/stanford-ai-experts-predict-what-will-happen-in-2026), 2026 — это год смещения от спекуляций к практической полезности. ИИ остаётся **ненадёжным и сильно зависимым от человеческого надзора**.

[MIT оценивает: 95% генеративных ИИ-пилотов не срабатывают](https://timspark.com/blog/why-ai-projects-fail-artificial-intelligence-failures-explained/) — часто из-за хрупких рабочих процессов и невыполненных ожиданий.

### Вывод

ИИ не замещает разработчиков. ИИ — это инструмент, как Git или Docker. Как любой инструмент, его нужно правильно использовать.

---

## 2. Где ИИ действительно полезен (и где нет)

### Где ИИ работает: узкие задачи

[Исследования показывают, что ИИ утраивает продуктивность примерно в трети задач](https://autofaceless.ai/blog/ai-productivity-statistics-2026):
- Написание черновиков (drafting)
- Исследование и анализ данных
- Генерация кода (в конкретных местах)
- Создание контента

**Постоянные пользователи получают 20+ часов экономии в неделю** (по Federal Reserve).

[Разработчики оценивают прирост ценности в 1.4–2x](https://metr.org/blog/2026-05-11-ai-usage-survey/), с прогнозом 2.5x к 2027 году.

### Где ИИ не работает: системный уровень

На макроэкономическом уровне всплеска производительности **не видно**. Если бы ИИ действительно экономил рабочую силу в масштабах, мы бы видели резкий рост производительности оставшихся рабочих. Его нет.

**Почему?** Потому что издержки на контроль качества ИИ-кода компенсируют выигрыш в скорости.

### Вывод

ИИ экономит время на конкретные куски работы. Но без правильной системы, вся экономия уходит на проверку и переделку.

## 3. Главная проблема: дисциплина

### Без guardrails: +35-40% ошибок

[Гарвардское исследование](https://blog.exceeds.ai/comparative-ai-code-quality-governance/): ИИ-генерируемый код содержит **в 1.7x больше ошибок**.

[Команды без guardrails качества](https://tfir.io/ai-code-quality-2026-guardrails/) сообщают об **увеличении плотности ошибок на 35-40% в течение 6 месяцев**.

Это не потому что ИИ плохой. Это потому что:
- ИИ-код "выглядит правильным" но содержит тонкие ошибки
- Опытные разработчики тратят на 19% больше времени на review
- Требуется глубокое понимание архитектуры, чтобы оценить решение

### С дисциплиной: реальная экономия

Команды с правильной дисциплиной (типизация, тесты, архитектура) получают реальный прирост производительности без роста ошибок.

### Вывод

**Дисциплина определяет всё.** ИИ без контроля — это способ быстро добавлять техдолг. ИИ с контролем — это инструмент для ускорения.

## 4. Фундамент: архитектура проекта

Чёткое структурирование проекта по файлам, папкам и паттернам — это **главный ключ к контролю написания кода**. С ИИ потребность в этом выросла критически.

ИИ пишет код как средний программист. И в этом его главная беда: он быстро создаёт файловый хаос, тысячи дублирующих методов, переменных. Всё мешается как в доме Оболенских.

В обычном проекте это происходит медленно, годами. С ИИ это произойдёт за **недели**. ИИ пишет очень быстро.

**Решение: явная архитектура заранее.**

### Выберите архитектурный подход

Существует несколько популярных подходов. Выберите один и следуйте ему. ИИ не должен выбирать — вы выбираете.

**Domain Driven Design (DDD)** — организация по бизнес-доменам:
```
src/domains/{users,products,orders}/
├── entities/
├── services/
├── repositories/
└── dtos/
```

**Clean Architecture** — организация по слоям:
```
src/
├── domain/
├── application/
├── infrastructure/
└── presentation/
```

**Feature-based** — организация по фичам:
```
src/features/{auth,payments,notifications}/
```

**Hexagonal Architecture** — организация по портам и адаптерам:
```
src/{core,ports,adapters}/
```

Каждый подход имеет плюсы и минусы. Выберите **один**, опишите его в CLAUDE.md, и требуйте следовать.

### Явные правила структуры

Напишите в CLAUDE.md:

```markdown
## Архитектура проекта

Используем Domain Driven Design.

### Правила:
- Каждый домен имеет папку `domains/{domain}/`
- Внутри: `entities/`, `services/`, `repositories/`, `dtos/`
- Никогда не создавайте новые папки без одобрения
- Если нужна новая папка — это сигнал нового домена
- Никогда не дублируйте код между доменами

### Зависимости (dependency-cruiser):
- Domain никогда не зависит от controllers
- Utils не может зависеть от services
```

### Dependency Injection — контроль скрытых зависимостей

Используйте DI чтобы ИИ не создавал скрытые зависимости:

```typescript
// ❌ ИИ часто пишет так (скрытая зависимость)
export class UserService {
  private db = new Database(); // Плохо!
}

// ✅ Правильно — зависимость явна
export class UserService {
  constructor(private db: Database) {}
}
```

С DI ИИ не может скрывать зависимости. Они видны в конструкторе.

### Инструменты для контроля архитектуры

Используйте `dependency-cruiser` для валидации правил:

```bash
npm install -D dependency-cruiser
npx depcruise src --validate
```

ИИ не сможет нарушить архитектуру — инструмент его остановит.

## 5. Процесс: Research-Plan-Implement

Перед тем как писать код, вы должны понимать что нужно делать. Используйте трёхэтапный процесс.

### 1. Research — исследование

Сначала вы сами исследуете:
- Какие файлы нужно изменить?
- Какой контекст существует?
- Как это согласуется с архитектурой?

Запросите у ИИ помощь в поиске файлов и понимании кода.

### 2. Plan — планирование

Напишите подробный **письменный план** (или спецификацию):

```
## План: добавить систему уведомлений

1. Создать domain notifications/ с entities, services, repositories
2. Interface NotificationService:
   - send(userId, message): Promise<void>
   - getHistory(userId): Promise<Notification[]>
3. Реализовать в services/
4. Создать HTTP endpoints в controllers/
5. Unit-тесты (80%+ покрытие)
6. Integration-тесты для критичных путей
```

**Или используйте спецификацию:**

```typescript
/**
 * Спецификация: NotificationService.send()
 * 
 * Входные данные:
 *   - userId: UUID
 *   - message: string (max 500 chars)
 * 
 * Выходные данные:
 *   - Notification с id, createdAt, status
 * 
 * Правила:
 *   - должно быть в очереди
 *   - retry при ошибке (макс 3 раза)
 * 
 * Ошибки:
 *   - InvalidUserError если userId не существует
 *   - MessageTooLongError если message > 500
 * 
 * Тесты:
 *   - успешная отправка
 *   - ошибка при невалидном userId
 *   - retry при сетевой ошибке
 */
```

**Ключевое правило: никогда не позволяйте ИИ писать код, пока не одобрите план.**

### 3. Implement — реализация

Только после одобрения плана запросите ИИ реализовать его:
- Пишет код по плану
- Не импровизирует
- Следует архитектуре

## 6. Контроль: типизация, тесты, архитектура

Когда ИИ пишет код, нужны трёхслойные фильтры.

### Слой 1: TypeScript strict mode

TypeScript отловит ошибки типов до runtime:

```typescript
// ИИ это сгенерирует — будет ошибка:
function processUser(user: User) {
  return user.email.toUpperCase(); // Error: possibly undefined
}

// ИИ будет вынужден исправить:
function processUser(user: User) {
  return user.email?.toUpperCase() ?? 'NO_EMAIL';
}
```

**Strict mode TypeScript не позволяет ИИ лениться.**

### Слой 2: Тесты (80%+ покрытие)

Даже если TypeScript не заметит, тесты отловят:

```typescript
// ИИ может написать это и TypeScript не заметит:
function discountPrice(price: number, percent: number): number {
  return price - (price * percent);  // Ошибка: не разделил на 100!
}

// Но тест сразу провалится:
it('should calculate 10% discount', () => {
  expect(discountPrice(100, 10)).toBe(90); // Fail!
});
```

**Требования:**
- Unit-тесты: 80%+ покрытия
- Integration-тесты: критичные пути
- Запуск перед каждым коммитом

### Слой 3: Code review

Финальная человеческая проверка:
- Соответствие плану
- Логические ошибки
- Edge cases
- Безопасность
- Техдолг

## 7. Оптимизация: скрипты и фичи ИИ

### Скрипты вместо постоянных запросов

Самая дорогая ошибка: каждый раз просить ИИ делать одно и то же.

**Вместо этого:** напишите скрипт один раз.

Примеры:
- Генератор миграций БД
- Codemod для рефакторинга по паттерну
- Скрипт обновления версий зависимостей
- Анализ логов и отчёты

**Правило:** если задача занимает > 1 минуты вручную и повторяется, напиши скрипт.

**Экономия:** один запрос ИИ (написать скрипт) вместо десяти (выполнить задачу 10 раз).

### Фичи ИИ в 2026

**Skills** — кастомные навыки для повторяющихся операций.

**Sub-agents** — несколько агентов работают параллельно на разных частях проекта.

**Prompt Caching** — кеширование контекста (экономия времени и денег на повторных запросах).

**Thinking Mode** — длинное мышление для сложного кода.

**Tool Use / MCP** — ИИ может вызывать инструменты прямо во время работы (запуск тестов, линтеров, проверка архитектуры).

**Batch Processing** — массовая обработка однотипных задач (50% дешевле).

## 8. Выводы: система работает

### Результат: реальная экономия без техдолга

Если вы следуете этой системе:

1. **Архитектура** — чёткая структура, dependency-cruiser
2. **Research-Plan-Implement** — контроль перед кодированием
3. **TypeScript strict** — автоматическая валидация типов
4. **Тесты 80%+** — фильтр логических ошибок
5. **Code review** — финальная проверка
6. **Скрипты для рутины** — экономия денег на повторных запросах
7. **Современные фичи ИИ** — Skills, Caching, Sub-agents

Вы получаете:
- **2-3x прирост производительности** без роста ошибок
- **Техдолг не растёт** — контроль на каждом этапе
- **Экономия денег** — меньше запросов благодаря скриптам и кешированию

### Как начать

1. Выберите архитектурный подход (DDD, Clean, Feature-based)
2. Опишите его в CLAUDE.md
3. Настройте dependency-cruiser для проверки правил
4. Требуйте Research-Plan перед каждым кодированием
5. 80%+ тесты обязательны
6. Используйте TypeScript strict

### Последний совет

Начните с малого. Не нужно всё внедрять сразу:

**Неделя 1:**
- Архитектура в CLAUDE.md
- Research-Plan процесс

**Неделя 2:**
- TypeScript strict mode
- 80%+ покрытие тестами

**Неделя 3+:**
- dependency-cruiser
- Скрипты для рутины
- Фичи ИИ (caching, skills)

Главное — **дисциплина и контроль**. Всё остальное придёт со временем.
