## Часть I: Архитектура и настройка

Чёткое структурирование проекта по файлам, папкам и паттернам — это **главный ключ к контролю написания кода**. С ИИ потребность в этом выросла критически.

ИИ пишет код как средний программист, и это его главная беда: он быстро создаёт файловый хаос, тысячи дублирующих методов, переменных. Всё мешается как в доме Оболенских.

В обычном проекте это происходит медленно, годами. С ИИ это произойдёт за **недели**. ИИ пишет очень быстро. Без явного контроля весь проект скатится в постоянно возрастающую сложность и технический долг намного быстрее.

**Решение: явная архитектура заранее.**

### Выберите архитектурный подход

Существует несколько подходов. Выберите один и следуйте ему. **ИИ не должен выбирать — вы выбираете.** Выбор зависит от типа проекта и его сложности.

#### Как выбрать?

**Для большинства проектов с ИИ: Feature-based или Clean Architecture** (легче контролировать AI).

Возможные архитектуры и когда их использовать:

**1. Feature-based** (рекомендуется для AI-проектов)
- **Структура:** `src/features/{feature_name}/{components,services,tests}`
- **Когда использовать:** новые проекты, средние по сложности приложения, когда требования четкие
- **Почему хорошо для ИИ:**
  - Структура очевидна из требований: "добавить фичу авторизация" → папка `features/auth`
  - AI трудно нарушить архитектуру — каждая фича изолирована
  - Спеки пишутся просто: "реализовать фичу X с компонентами A, B, C"
  - Минимум контекста о доменах/бизнесе нужно
- **Пример:** e-commerce = features/products, features/cart, features/checkout, features/auth

**2. Clean Architecture** (хорошо для ИИ)
- **Структура:** слои `entities → use-cases → interface-adapters → frameworks`
- **Когда использовать:** когда нужна высокая изолированность, сложная бизнес-логика
- **Почему хорошо для ИИ:**
  - Слои четко разделены, AI трудно их нарушить
  - Типичный паттерн: controller → service → repository
  - Зависимости направлены внутрь (контролируемо)
  - Тесты пишутся четко: unit (entities), integration (use-cases)
- **Минус:** требует больше кода, структура менее очевидна из требований

**3. Domain Driven Design (DDD)**
- **Структура:** `domains/{domain}/{entities,services,repositories,dtos}`
- **Когда использовать:** большие корпоративные системы с четкими бизнес-доменами
- **Почему требует осторожности с ИИ:**
  - Требует глубокого понимания бизнес-логики и термином (bounded contexts, ubiquitous language)
  - AI пишет "как средний программист" (не стратегически)
  - DDD требует постоянного рефакторинга и пересмотра доменов
  - Легче создать неправильные домены, которые потом переделывать
  - Нужна опытная команда, которая направляет AI
- **Когда может работать:** если у вас есть Product Owner или архитектор, который четко определяет домены

**4. Hexagonal Architecture** (портов и адаптеров)
- **Структура:** core → ports (interfaces) → adapters (реализация)
- **Когда использовать:** когда нужна максимальная гибкость, частая смена деталей реализации
- **Почему может быть сложна для ИИ:** требует понимания разницы между портом и адаптером, AI может их перепутать

**5. Microservices Architecture**
- **Когда использовать:** большие команды, разные сервисы развивают независимо
- **Для AI:** сложно начинать с этого, лучше начать с Feature-based, потом разбить на микросервисы

**6. Event-Driven Architecture**
- **Когда использовать:** real-time системы, асинхронные процессы, много событий
- **Для AI:** может быть сложна, требует понимания потока событий

#### Рекомендация для начинающих

Начните с **Feature-based**, если:
- Проект новый
- Требования понятные
- Команда маленькая (1-5 людей)
- Нужна скорость разработки

Начните с **Clean Architecture**, если:
- Нужна высокая тестируемость
- Бизнес-логика сложная
- Надо часто менять детали реализации

Используйте **DDD** только если:
- Вы уже использовали его раньше
- У вас есть архитектор, который направляет команду
- Домены четко определены в требованиях

Полные описания, примеры и рекомендации смотрите в [документации архитектур](https://github.com/tertiumnon/tertium-cheatsheets/tree/main/pages/arhitectures).

### CLAUDE.md — инструкции для ИИ

Это одна из самых недооцененных вещей в работе с ИИ. **CLAUDE.md — это единственный файл, который вы пишете один раз, а ИИ читает его в каждом запросе.**

#### Зачем он нужен?

- **ИИ всегда знает правила** вашего проекта (не нужно повторять)
- **Prompt Caching** кеширует его содержимое (экономия денег и времени)
- **Единственный источник истины** о том как писать код
- **Защита от ошибок** — все правила и ограничения явно указаны

#### Что писать в CLAUDE.md?

Пример полного CLAUDE.md для типичного проекта:

```markdown
# CLAUDE.md - Инструкции для ИИ

## 1. Архитектура проекта

Используем **Feature-based архитектуру**.

### Правила структуры:
- Каждая фича в папке `src/features/{feature_name}/`
- Внутри каждой фичи: `components/`, `services/`, `types/`, `tests/`
- Никогда не создавайте новые папки на уровне src/
- Если нужна новая папка вне фичи — это требует обсуждения

### Примеры:
```
src/
├── features/
│   ├── auth/
│   │   ├── components/
│   │   ├── services/
│   │   └── tests/
│   ├── products/
│   ├── cart/
│   └── checkout/
├── shared/
│   ├── components/
│   ├── utils/
│   └── types/
```

## 2. Технологический стек

- **Runtime:** Node.js v20+
- **Language:** TypeScript с `"strict": true`
- **Framework:** Angular v20
- **Testing:** Vitest (NOT Jest)
- **Package manager:** Bun (NOT npm or yarn)
- **Linting:** Rome (NOT ESLint)
- **Database:** PostgreSQL (если есть)

## 3. Соглашения кода

### TypeScript:
- Всегда `"strict": true` в tsconfig.json
- Никогда не используй `any` (используй `unknown` если нужно)
- Экспортируй только необходимое (default exports запрещены)
- Один компонент = один файл

### Файлы:
- Используй kebab-case для файлов: `user-service.ts`, NOT `userService.ts`
- Максимум 300 строк на файл (если больше — разбей на части)
- Комментарии только для WHY, не для WHAT

### Функции:
- Максимум 20 строк на функцию
- Одна функция = одна ответственность
- Обязательно типизируй параметры и возвращаемое значение

### Тесты:
- 80%+ покрытие обязательно
- Файл .test.ts рядом с исходным файлом
- Используй `describe` и `it` блоки

## 4. CI/CD и Deployment

- Сборка: `bun run build`
- Тесты: `bun test`
- Линтинг: `bun run lint:fix`
- Type check: `bun run lint:ts:check`
- Deploy: `bun run deploy`

Все скрипты находятся в `package.json`.

## 5. Если что-то не ясно

Правило: **Требуй плана перед кодированием**

Если запрос неясен:
1. Попроси уточнение
2. Напиши план что будешь делать
3. Жди подтверждения перед кодированием

Пример:
```
Вы: "Добавь систему уведомлений"

ИИ должен ответить:
- Где хранить уведомления? (В памяти? В БД? В очереди?)
- Как их отправлять? (WebSocket? Email? Push?)
- Какой интерфейс? (Компонент? Service? Обе?)

Жди ответа перед кодированием.
```

## 6. Главное правило

**Контроль > Скорость**

Лучше медленно и правильно, чем быстро и с ошибками.
```

#### Как использовать CLAUDE.md в работе

1. **Создай файл** `CLAUDE.md` в корне проекта
2. **Поделись с ИИ:** "Вот мой CLAUDE.md, соблюдай эти правила"
3. **При Prompt Caching:** ИИ автоматически будет кешировать эти правила
4. **Обновляй при необходимости:** если меняются правила проекта

#### Когда обновлять CLAUDE.md

- Когда архитектура меняется
- Когда появляются новые запрещённые команды
- Когда меняется tech stack
- Когда вы заметили что ИИ часто ошибается в одном месте

**Совет:** Каждый раз когда ИИ ошибается по одной и той же причине — добавьте это правило в CLAUDE.md.

### Dependency-cruiser — валидация архитектуры

Используйте `dependency-cruiser` для валидации правил:

```bash
npm install -D dependency-cruiser
npx depcruise src --validate
```

ИИ не сможет нарушить архитектуру — инструмент его остановит.

Напишите правила в CLAUDE.md:

```markdown
## Архитектура проекта

Используем Domain Driven Design.

### Правила:
- Каждый домен имеет папку `domains/{domain}/`
- Внутри: `entities/`, `services/`, `repositories/`, `dtos/`
- Никогда не создавайте новые папки без одобрения
- Если нужна новая папка — это сигнал нового домена
- Никогда не дублируйте код между доменами
- Максимум 500 строк на файл

### Зависимости (dependency-cruiser):
- Domain никогда не зависит от controllers
- Utils не может зависеть от services
- Repositories не пишут бизнес-логику
```


## Часть II: Технологии и инструменты

ИИ работает по-разному на разных языках. Выбор языка влияет на скорость разработки и количество ошибок. В этой части собраны все рекомендации по языкам: типизация, тестирование, линтеры и особенности работы с ИИ.

### TypeScript/JavaScript

**Лучший выбор для AI-разработки.**

**Почему хорошо:** динамическая типизация позволяет ИИ быстро писать, но TypeScript strict mode ловит ошибки.

**Риск:** если не включить strict mode, ИИ создаст много скрытых ошибок типов.

#### Типизация (strict mode)

Включите в `tsconfig.json`:
```json
{
  "compilerOptions": {
    "strict": true,
    "noImplicitAny": true,
    "strictNullChecks": true,
    "strictFunctionTypes": true
  }
}
```

```typescript
// ❌ ИИ это сгенерирует — будет ошибка:
function processUser(user: User) {
  return user.email.toUpperCase(); // Error: Object is possibly 'undefined'
}

// ✅ ИИ будет вынужден это исправить:
function processUser(user: User) {
  return user.email?.toUpperCase() ?? 'NO_EMAIL';
}
```

#### Тестирование

**Инструменты:** Vitest (рекомендуется) или Jest

```typescript
// ❌ ИИ может написать это и type checking не заметит:
function discountPrice(price: number, percent: number): number {
  return price - (price * percent);  // Ошибка: не разделил на 100!
}

// ✅ Но тест сразу провалится:
it('should calculate 10% discount correctly', () => {
  expect(discountPrice(100, 10)).toBe(90); // Fail!
});
```

Запуск с покрытием: `npm run test -- --coverage`

#### Линтеры и форматтеры

- ESLint или Rome для статического анализа
- Prettier для форматирования

#### Dependency Injection

```typescript
// ❌ ИИ часто пишет так (скрытая зависимость)
export class UserService {
  private db = new Database(); // Создание здесь — плохо!

  async createUser(name: string) {
    return this.db.query('INSERT INTO users...');
  }
}

// ✅ Правильно — зависимость явна в конструкторе
export class UserService {
  constructor(private db: Database) {}

  async createUser(name: string) {
    return this.db.query('INSERT INTO users...');
  }
}
```

#### Примеры хорошего и плохого кода

```typescript
// ✅ Явные типы, явные зависимости, единственная ответственность
export class UserService {
  constructor(private db: Database) {}

  async createUser(name: string): Promise<User> {
    if (!name || name.length < 2) {
      throw new Error("Invalid name");
    }
    return this.db.users.create({ name });
  }
}
```

```typescript
// ✅ Простой компонент с явной логикой
@Component({
  selector: 'app-user-form',
  standalone: true,
  template: `<form (ngSubmit)="onSubmit()">...</form>`
})
export class UserFormComponent {
  form = new FormGroup({ name: new FormControl('') });

  constructor(private service: UserService) {}

  onSubmit(): void {
    this.service.createUser(this.form.value.name);
  }
}
```

```typescript
// ❌ Скрытые зависимости
class UserService {
  private db = new Database(); // Плохо!
}

// ❌ Слишком большая функция
async function handleEverything() { /* 500 строк */ }

// ❌ Нарушение архитектуры
// компонент пишет прямо в БД вместо использования service
```

### Python

**Хорошо для бизнес-логики и ML.**

**Почему хорошо:** синтаксис простой, ИИ понимает его хорошо, быстрая разработка.

**Риск:** runtime errors, динамическая типизация может привести к ошибкам.

**Где использовать:** backend логика, ML/Data Science, скрипты.

#### Типизация (mypy + type hints)

```python
# ❌ Без type hints ИИ создаст ошибки:
def process_user(user):
    return user.email.upper()  # Может быть None, не проверено

# ✅ С type hints + mypy проверит:
from typing import Optional

def process_user(user: User) -> str:
    if user.email is None:
        return 'NO_EMAIL'
    return user.email.upper()
```

Включите `mypy` в проект:
```bash
mypy --strict src/
```

**Совет:** пишите docstrings для функций (ИИ лучше понимает контекст).

#### Тестирование

**Инструменты:** pytest + pydantic для валидации

```python
# ❌ ИИ может написать это:
def discount_price(price: float, percent: float) -> float:
    return price - (price * percent)  # Ошибка: не разделил на 100!

# ✅ Тест отловит:
def test_discount_price():
    assert discount_price(100, 10) == 90  # Fail!
```

Запуск с покрытием: `pytest --cov`

#### Dependency Injection

```python
# ❌ Скрытая зависимость
class UserService:
    def __init__(self):
        self.db = Database()  # Плохо!

    def create_user(self, name: str):
        return self.db.query('INSERT INTO users...')

# ✅ Явная зависимость
class UserService:
    def __init__(self, db: Database):  # Зависимость видна
        self.db = db

    def create_user(self, name: str):
        return self.db.query('INSERT INTO users...')
```

#### Линтеры и форматтеры

- black для форматирования
- flake8 или ruff для линтинга

### Go

**Хорошо для backend и микросервисов.**

**Почему хорошо:** строгая типизация, простой синтаксис, быстрая компиляция.

**Риск:** ИИ может забыть проверки ошибок (`if err != nil`).

**Где использовать:** backend, микросервисы, CLI инструменты.

#### Типизация (встроенная)

```go
// Go требует явных типов, ИИ не может это нарушить
func ProcessUser(user User) string {
    if user.Email == "" {
        return "NO_EMAIL"
    }
    return strings.ToUpper(user.Email)
}
```

**Совет:** обязательны code review на все файлы (проверяйте обработку ошибок).

#### Тестирование

**Инструменты:** встроенный testing пакет

```go
// ❌ ИИ может написать это:
func DiscountPrice(price float64, percent float64) float64 {
    return price - (price * percent)  // Ошибка!
}

// ✅ Тест отловит:
func TestDiscountPrice(t *testing.T) {
    result := DiscountPrice(100, 10)
    if result != 90 {
        t.Errorf("expected 90, got %v", result)
    }
}
```

Запуск с покрытием: `go test -cover`

#### Dependency Injection

```go
// ❌ Скрытая зависимость
type UserService struct {
    db *Database  // Если инициализируется внутри — плохо
}

// ✅ Явная зависимость
type UserService struct {
    db *Database
}

func NewUserService(db *Database) *UserService {  // Зависимость видна
    return &UserService{db: db}
}
```

#### Линтеры и форматтеры

**Почему хорошо:** строгая типизация, явные интерфейсы помогают ИИ.

**Риск:** многословие, ИИ может генерировать слишком много boilerplate.

**Совет:** используйте Kotlin вместо Java (более лаконичный).

fun processUser(user: User): String {

**Инструменты:** JUnit

// ✅ Тест отловит:

#### Dependency Injection

```java
// ❌ Скрытая зависимость
public class UserService {

        return db.query("INSERT INTO users...");
    }
}

// ✅ Явная зависимость (через конструктор или setter)
public class UserService {
    private Database db;

    public UserService(Database db) {  // Зависимость видна
        this.db = db;

## Часть III: Процесс разработки

### Research-Plan-Implement

Перед тем как писать код, вы должны понимать что нужно делать. Используйте трёхэтапный процесс.

#### 1. Research — исследование

Сначала вы сами исследуете:
- Какие файлы нужно изменить?
- Какой контекст уже существует?
- Как это согласуется с архитектурой?
- Какие ограничения и соглашения нужно учесть?

Запросите у ИИ помощь в поиске нужных файлов и понимании кода. На этом этапе ИИ просто читает и объясняет, не пишет.

#### 2. Plan — планирование

Напишите подробный **письменный план** (или спецификацию):

```
## План: добавить систему уведомлений

1. Создать domain notifications/ с:
   - entities/Notification.ts
   - services/NotificationService.ts
   - repositories/NotificationRepository.ts

2. NotificationService должен иметь:
   - send(userId, message): Promise<Notification>
   - getHistory(userId): Promise<Notification[]>

3. Реализовать в infrastructure/ для БД

4. HTTP endpoints в controllers/NotificationController

5. Unit-тесты (минимум 80% покрытие):
   - успешная отправка
   - обработка ошибок
   - history retrieval

6. Integration-тесты для критичных путей
```

**Или используйте спецификацию:**

```typescript
/**
 * Спецификация: NotificationService.send()
 *
 * Входные данные:
 *   - userId: UUID (должен существовать)
 *   - message: string (max 500 chars)
 *
 * Выходные данные:
 *   - Notification { id, userId, message, createdAt, status }
 *
 * Правила бизнеса:
 *   - должно быть в очереди (не сразу отправляется)
 *   - retry при сетевой ошибке (макс 3 раза)
 *   - логирование всех попыток
 *
 * Ошибки (должны быть обработаны):
 *   - InvalidUserError если userId не существует
 *   - MessageTooLongError если message > 500
 *   - QueueError если очередь недоступна
 *
 * Тесты (обязательный минимум):
 *   - успешная отправка
 *   - ошибка при невалидном userId
 *   - retry при сетевой ошибке
 *   - сообщение > 500 chars отклонено
 */
```

**Ключевое правило: никогда не позволяйте ИИ писать код, пока не одобрите план.**

#### 3. Implement — реализация

Только после одобрения плана запросите ИИ реализовать его:
- Пишет код по плану
- Не импровизирует
- Следует архитектуре

Это гарантирует:
- Вы контролируете архитектурные решения до их реализации
- ИИ работает по вашему плану, а не импровизирует
- Проще отловить проблемы на этапе планирования, чем после кодирования
- Экономия времени на переделку неправильного подхода

### Git workflow

#### Commit message формат

```
type: short description

- More details if needed
- Second point

Co-Authored-By: Claude <noreply@anthropic.com>
```

#### Типы (type)

- `feat:` новая фича
- `fix:` исправление баги
- `docs:` документация
- `refactor:` рефакторинг
- `test:` добавление тестов
- `ci:` CI/CD изменения

#### Правила

- Никогда не коммитьте в main без PR
- PR должен пройти CI перед merge
- Требуется одобрение перед merge (если есть code review)
- Используй `git add [specific files]`, NOT `git add .`

#### Паттерн: ИИ может коммитить много раз, потом объединить

**Проблема:** ИИ часто коммитит после каждого небольшого изменения. Это приводит к 50+ коммитов на простую фичу.

**Решение:** Разрешь ИИ коммитить часто (это помогает отслеживать прогресс), но в конце объедини коммиты.

**Во время разработки (рабочая ветка):**
```bash
# ИИ делает много маленьких коммитов
feat: add user service
fix: handle null case in user service
test: add tests for user service
refactor: improve error handling in user service
docs: add documentation for user service
```

**В конце (перед PR):**
```bash
# Объедини все в один-два логичных коммита
git rebase -i HEAD~5  # интерактивный rebase последних 5 коммитов
```

Выбери первый коммит как `pick`, остальные как `squash` (s):
```
pick feat: add user service
s fix: handle null case in user service
s test: add tests for user service
s refactor: improve error handling
s docs: add documentation
```

**Результат:** один коммит `feat: add user service` со всеми изменениями

#### Когда использовать этот паттерн

**Используй:**
- Для новых фич (объедини в 1 коммит)
- Для багфиксов (1 коммит)
- Для рефакторинга (1-2 коммита)

**НЕ используй:**
- Для разных независимых фич (каждая должна быть отдельным коммитом)
- Если уже pushed в remote (не делай rebase после push)

#### Инструкция для ИИ в CLAUDE.md

```markdown
### Коммиты

ИИ может коммитить часто (после каждого логического шага):
- После добавления service
- После написания тестов
- После рефакторинга

**Но перед PR:** объедини коммиты одной фичи в 1-2 коммита:
```bash
git rebase -i HEAD~N  # N = количество коммитов
# Mark first as pick, rest as squash
```

Каждый коммит должен быть самодостаточным (проходит тесты).
```

#### Почему это хорошо

- **Для разработки:** ИИ может отслеживать прогресс, коммитить шаг за шагом
- **Для истории:** чистая история без 50 "fix typo" коммитов
- **Для Code Review:** проще просматривать полностью готовую фичу, чем 50 маленьких PR
- **Для Prompt Caching:** ИИ кеширует контекст, экономит деньги, работает быстрее

### Контекст и Workspace

#### Проблема выбора

Когда вы работаете с ИИ, встаёт вопрос: **нужно ли создавать новую сессию для каждой задачи или лучше продолжить одну?**

Ответ: **Зависит от типа работы.**

#### Вариант 1: Один непрерывный контекст (Рекомендуется для проекта)

**Когда использовать:** когда вы работаете над одним проектом целый день/неделю.

**Как это работает:**
- Открываете workspace проекта один раз
- Используете одну сессию ИИ целый день
- Все задачи идут в одном контексте: "добавь фичу X", потом "исправь баг Y", потом "добавь тесты Z"
- ИИ видит весь проект, его архитектуру, соглашения, `CLAUDE.md`

**Преимущества:**
- **Контекст сохраняется** — ИИ помнит архитектуру, стиль, паттерны
- **Prompt Caching** — `CLAUDE.md` и структура проекта кешируются (экономия денег и скорость)
- **Меньше повторений** — не нужно каждый раз объяснять архитектуру
- **Консистентность** — ИИ пишет код в одном стиле целый день
- **Ошибки ловятся быстро** — если ИИ нарушил паттерн, вы видите это сразу

**Недостатки:**
- Контекстное окно растёт (но Prompt Caching помогает)
- Если большой файл был обсуждён — остаётся в памяти

**Рекомендация:** Используйте это для большинства случаев.

#### Вариант 2: Новый контекст для каждой задачи

**Когда использовать:** когда задачи полностью независимые или вы меняете проекты.

**Как это работает:**
- Вы закрываете предыдущую сессию
- Для новой задачи открываете новый контекст
- ИИ читает `CLAUDE.md`, смотрит затронутые файлы, и решает задачу

**Преимущества:**
- Чистый контекст (нет "мусора" от предыдущих задач)
- Меньше контекстного окна (для больших проектов)
- Каждая задача решается свежо

**Недостатки:**
- **Без Prompt Caching** — `CLAUDE.md` читается каждый раз (медленнее, дороже)
- **Потеря контекста архитектуры** — нужно каждый раз напоминать
- **Риск нарушения паттернов** — ИИ может забыть соглашения
- **Больше ошибок** — нет "памяти" о том как вы делали прошлые фичи

**Рекомендация:** Не рекомендуется. Используйте только если контекст действительно переполняется (редко с Prompt Caching).

#### Лучшая практика: Гибридный подход

**Используйте один контекст в течение дня**, но:

1. **Каждый день — новая сессия** (вечером закрываете, утром открываете новую)
   - Это очищает контекст, но `CLAUDE.md` всё равно кешируется

2. **Если контекст стал слишком большим** — зафиксируйте прогресс коммитом и начните новую сессию
   - `git commit` with descriptive message
   - Закройте сессию
   - Откройте новую с тем же workspace
   - ИИ по-прежнему видит весь проект (через файлы), но контекст чистый

3. **Между разными проектами** — всегда новая сессия
   - Закрывайте предыдущий проект
   - Открывайте новый workspace
   - Новый контекст

#### Workspace и Full-Stack разработка

**Workspace** в Claude Code позволяет ИИ видеть весь проект одновременно.

Когда вы открываете workspace полного стека (frontend + backend):

```
project/
├── backend/           # Node.js + Express
│   ├── src/
│   ├── package.json
│   └── .env
├── frontend/          # Angular/React
│   ├── src/
│   ├── angular.json
│   └── package.json
├── CLAUDE.md          # Единые правила
└── README.md
```

**Что ИИ видит и может делать:**

1. **Видит архитектуру обеих сторон** — backend структуру И frontend структуру
2. **Может менять backend и frontend в одной задаче** — "добавь поле user.role в БД, в API endpoint, и покажи его в профиле"
3. **Консистентность между слоями** — ИИ может проверить что типы совпадают: backend возвращает `User{ id, name, role }`, frontend ожидает то же
4. **Видит конфигурацию обеих** — tsconfig, package.json, build scripts

**Преимущества для full-stack:**
- Одна задача = полный цикл (DB → API → UI)
- Меньше ошибок типов между слоями
- ИИ может проверить что backend и frontend согласованы

**Пример: добавить новый API endpoint с full-stack контекстом**

```
Вы: "Добавь поле 'premium' в user профиль. Backend: добавь в БД,
создай миграцию, добавь в API. Frontend: покажи это поле в UI"

ИИ видит:
- Backend: структура User entity, repository, controller
- Frontend: структура User interface, component, service
- CLAUDE.md: как писать код для обеих сторон
- Может сделать всё за один запрос
```

#### Когда НЕ использовать workspace для full-stack

- **Очень большой проект (50K+ строк)** — слишком много контекста
- **Разные команды backend/frontend** — лучше использовать разные workspace
- **Часто меняются одна сторона** — может быть лишняя информация

**Рекомендация:** Для большинства full-stack проектов — **один workspace, один контекст, весь день.**

#### Оптимальная workflow для full-stack разработки

1. **Утром:** открываете IDE с workspace проекта
2. **Целый день:** все задачи идут в одном контексте ИИ
3. **Вечером:** коммитите прогресс, закрываете IDE
4. **Завтра утром:** открываете снова (новая сессия, но project видим весь)

**Результат:**
- Контекст чистый каждый день
- Architekture кешируется (Prompt Caching)
- Full-stack консистентность
- Скорость и качество оптимальны


## Часть IV: Контроль качества

### Code review

**Финальная человеческая проверка.** Даже с типизацией и тестами нужен человеческий взгляд.

Проверьте:
- Соответствие плану
- Логические ошибки (которые не поймала типизация и тесты)
- Edge cases в тестах
- Безопасность (нет ли уязвимостей)
- Обработка ошибок (особенно важно для Go, Java)
- Соответствие соглашениям проекта
- Техдолг (не создал ли новые проблемы)

### CI/CD интеграция

Автоматизируйте проверки качества:

```yaml
# Пример GitHub Actions
- name: Type check
  run: bun run lint:ts:check

- name: Lint
  run: bun run lint:fix

- name: Tests
  run: bun test --coverage

- name: Build
  run: bun run build
```

`deploy` в CI-гейт обычно не входит — запускается вручную или отдельным pipeline после того, как все проверки выше прошли зелёным. Полный список скриптов — в `package.json` (пример CLAUDE.md из Части I уже показывает, как их описать для ИИ).

### Принципы тестирования

**Главный фильтр** — тесты. Даже если type checking не заметит, тесты отловят логические ошибки.

**Что ловят тесты:**
- Граничные случаи (пустые массивы, нулевые значения, большие числа)
- Состояния, которые не должны быть достижимы
- Взаимодействие между компонентами
- Race conditions и асинхронные ошибки
- Логику, которую type checking не может проверить

**Требования:**
- **Unit-тесты**: 80%+ покрытия кода
- **Integration-тесты**: критичные пути (авторизация, оплата, сохранение данных)
- **Snapshot-тесты**: для UI компонентов

**Универсальное правило:** Типизация не позволяет ИИ лениться. Это больше, чем проверка синтаксиса — это **автоматическая валидация всей логики**, независимо от языка.


## Часть V: Безопасность

Это реальная проблема, которую часто не замечают при работе с ИИ. ИИ может **непреднамеренно** выполнить деструктивные команды:

- `rm -rf .git` → удаление репозитория
- `rm -rf /` → удаление файловой системы (на производстве это чаще `docker exec`)
- `DROP TABLE users;` → удаление таблицы в БД
- `DELETE FROM *` → удаление всех данных
- Перезапись production переменных на development значения

**Это случилось со мной.** Я попросил ИИ написать скрипт очистки, и он создал `rm -rf *` на gh-pages orphan ветке. Это привело к удалению `.git` директории, и репозиторий пришлось восстанавливать из облака.

### Типичные сценарии ущерба

1. **Скрипты миграции и очистки:** "напиши скрипт удаления старых файлов" → `rm -rf *` вместо `rm -rf src/old_files/`

2. **Deployment скрипты:** "развёрнуть новую версию" → ИИ пишет команду которая останавливает production БД вместо staging

3. **Тестовые данные:** "очисти тестовую БД" → ИИ очищает production БД (перепутал переменные окружения)

4. **Управление зависимостями:** "удали неиспользуемый пакет" → ИИ удаляет критически важный пакет, который используется через `require()`

5. **Миграции БД:** "добавь индекс" → ИИ пишет миграцию которая дропает таблицу вместо её модификации

### Стратегии защиты

#### 1. Запрещённые команды в CLAUDE.md

ИИ НИКОГДА не должен выполнять эти команды без явного разрешения:

```bash
# Файловая система
rm -rf /
rm -rf .
rm -rf * (без указания конкретной папки)

# База данных
DROP TABLE
DROP DATABASE
DELETE FROM (без WHERE условия)

# Git
git reset --hard
git push --force
git rebase -i

# Системные
npm install -g
sudo [anything]
```

**Правило:** Если ИИ предлагает такую команду — всегда требуйте подробное объяснение.

#### 2. Операции требующие обязательного REVIEW

Эти команды должны быть явно согласованы перед выполнением:

- Любой `rm`, `delete`, `drop` (даже на test)
- Изменения переменных окружения
- Изменения скриптов deploy/backup
- Изменения конфигурации БД
- Изменения правил Git (commits, pushes)
- Создание новых сервисов или инфраструктуры
- Бизнес-логика, которая затрагивает данные пользователей

**Правило:** Если сомневаешься — требуй review.

#### 3. Никогда не давайте ИИ прямой доступ к destructive командам

```bash
# ПЛОХО: ИИ может написать опасную команду
bun run clean  # ИИ пишет rm -rf внутри

# ХОРОШО: Явный контроль
bun run clean:dev      # удаляет только dev файлы
bun run clean:tests    # удаляет только тестовые артефакты
bun run db:reset:staging  # сброс ТОЛЬКО staging БД
```

#### 5. Автоматическая защита в скриптах

```typescript
// Убедитесь что скрипт знает что он делает
async function deleteOldFiles(dir: string) {
  // Явная защита
  if (!dir.includes("temp") && !dir.includes("cache")) {
    throw new Error(`Cannot delete from ${dir}! Only temp/ and cache/ allowed`);
  }

  const files = await fs.readdir(dir);

  // Еще одна защита: не удалять слишком много
  if (files.length > 10000) {
    throw new Error("Too many files! This looks like a mistake");
  }

  console.log(`Deleting ${files.length} files from ${dir}...`);
  for (const file of files) {
    await fs.unlink(path.join(dir, file));
  }
}
```

## Часть VI: Оптимизация

Части I–V — про контроль: архитектура, типизация, тесты, безопасность. Здесь — про деньги: где расходы на ИИ утекают незаметно и что с этим делать на практике.

### Скрипты вместо постоянных запросов

Самая дорогая ошибка: **каждый раз просить ИИ делать одно и то же**.

**Вместо этого:** напишите скрипт один раз, потом запускайте его всегда.

Примеры:
- **Генератор миграций БД** — вместо просьбы ИИ писать SQL каждый раз
- **Codemod для рефакторинга** — вместо просьбы переписать компоненты вручную
- **Скрипт обновления версий** — вместо просьбы обновить dependencies каждый месяц
- **Анализ логов и отчёты** — вместо просьбы проанализировать данные каждый день

**Экономия:**
- Один запрос ИИ (написать скрипт) вместо десяти повторных запросов (выполнить задачу вручную 10 раз). **«В 10 раз дешевле» — это моя собственная прикидка для типичного случая, а не измеренный где-либо бенчмарк**: реальная экономия зависит от сложности задачи, модели и того, сколько раз скрипт реально переиспользуется — при однократном использовании скрипт может выйти даже дороже одного запроса
- Скрипт работает детерминированно — одинаковый результат каждый раз, в отличие от повторной генерации кода моделью
- Можно запустить по расписанию (cron, GitHub Actions) без участия ИИ вообще

**Правило большого пальца:** если задача занимает > 1 минуты вручную и повторяется, напиши скрипт.

### Выбор модели под задачу

Не все запросы стоят одинаково — и не все требуют одинаково мощной модели. Официальная страница ценообразования Anthropic называет это первой стратегией оптимизации расходов: [«выбирайте подходящую модель — Haiku для простых задач, Sonnet для большинства продакшн-нагрузок, Opus для самых сложных рассуждений»](https://platform.claude.com/docs/en/about-claude/pricing#additional-pricing-considerations).

Разница в цене — не мелочь. По состоянию на середину 2026 года Claude Haiku 4.5 стоит $1 / $5 за миллион входных/выходных токенов, Claude Sonnet 5 — $2 / $10 по вводной цене (действует до 31 августа 2026 года, дальше $3 / $15), Claude Opus 4.8 — $5 / $25. Haiku в несколько раз дешевле Sonnet и в 5 раз дешевле Opus — [полная таблица цен на модели](https://platform.claude.com/docs/en/about-claude/pricing#model-pricing).

[Официальный гайд Claude Code по управлению расходами](https://code.claude.com/docs/en/costs#choose-the-right-model) формулирует это как практическое правило: «Sonnet справляется с большинством задач кодирования и стоит дешевле Opus. Оставьте Opus для сложных архитектурных решений и многошаговых рассуждений». Там же — конкретная рекомендация для мультиагентной работы (см. [«Sub-agents и мультиагентная оркестрация»](#sub-agents-и-мультиагентная-оркестрация) в Части VII): для простых воркеров можно явно задать `model: haiku` в конфигурации субагента, не тратя Sonnet или Opus на механическую работу вроде поиска файлов или прогона тестов.

Ещё один рычаг — не переключать модель целиком, а тюнить effort в рамках одной модели. [Официальная документация по выбору модели](https://platform.claude.com/docs/en/about-claude/models/choosing-a-model) отмечает, что последние модели Opus и Sonnet поддерживают параметр effort, который меняет баланс интеллект/задержка/цена без смены модели — и часто это более удобный рычаг, чем переключение между Sonnet и Opus.

### Гигиена контекста: экономия на очистке

[«Контекст и Workspace»](#контекст-и-workspace) в Части III разбирает, *когда* открывать новую сессию — это вопрос рабочего процесса. Здесь тот же вопрос под другим углом: почему раздутый контекст напрямую бьёт по счёту, а не только по качеству ответов.

Токены масштабируются с размером контекста: чем больше контекста обрабатывает модель, тем больше вы платите — причём за *каждое* следующее сообщение в сессии, не только за то, что раздуло контекст. [Официальный гайд Claude Code по управлению расходами](https://code.claude.com/docs/en/costs#when-a-developer-asks-about-a-limit) прямо называет это типичной причиной неожиданно больших счетов: «необычно высокие расходы на API или облачном тарифе обычно объясняются долгими сессиями, которые никогда не чистили, или Opus, оставленным моделью по умолчанию». Практическая рекомендация оттуда же: используйте `/clear` между несвязанными задачами — «устаревший контекст тратит токены на каждом следующем сообщении».

Автосуммаризация (compaction, `/compact`) смягчает проблему, но не отменяет её: [документация по compaction](https://platform.claude.com/docs/en/build-with-claude/compaction) уточняет, что сама суммаризация — это дополнительный вызов модели, который добавляет токены сейчас, чтобы предотвратить их неограниченный рост дальше по сессии. Иначе говоря, `/clear` между независимыми задачами почти всегда дешевле, чем дать контексту накопиться и потом его сжимать. (Экономия на CLAUDE.md за счёт Prompt Caching при этом никуда не девается — она разобрана в [Части VII](#инфраструктурные-фичи-2026).)

### Точные промпты вместо широкого сканирования

Расплывчатый запрос заставляет ИИ читать файлы, чтобы понять, что вообще происходит, — и это тоже токены. [Официальный гайд Claude Code](https://code.claude.com/docs/en/costs#write-specific-prompts) формулирует это прямо: «расплывчатые запросы вроде "улучши эту кодовую базу" запускают широкое сканирование. Конкретные запросы вроде "добавь валидацию input в функцию login в auth.ts" позволяют Claude работать эффективно, читая минимум файлов».

Тот же принцип работает не только для промптов, но и для инструментов:

- **CLI вместо MCP, когда возможно.** [По тому же гайду](https://code.claude.com/docs/en/costs#reduce-mcp-server-overhead), CLI-инструменты (`gh`, `aws`, `gcloud`) экономичнее по контексту, чем MCP-серверы, — они не добавляют в контекст описание каждого доступного инструмента, Claude просто запускает команду напрямую
- **Хуки для предобработки шумного вывода.** Вместо того чтобы отдавать ИИ лог-файл на 10 000 строк, хук может сам прогнать `grep ERROR` и вернуть только совпадения — [пример из документации](https://code.claude.com/docs/en/costs#offload-processing-to-hooks-and-skills) сокращает контекст с десятков тысяч токенов до сотен
- **Skills вместо разведки с нуля.** Skill с описанием архитектуры проекта («codebase-overview») отдаёт ИИ нужный контекст сразу, вместо того чтобы тот тратил токены на чтение файлов ради понимания структуры — подробнее про Skills в [Части VII](#skills--переиспользуемые-навыки)

Это тот же принцип, что и в [Research-Plan-Implement](#research-plan-implement) из Части III: чем точнее поставлена задача заранее, тем меньше ИИ тратит на угадывание. Разница только в том, что здесь речь не о качестве результата, а о счёте за токены.


## Часть VII: Подходы к работе с ИИ

Части I–VI описывают, *что* настроить в проекте, чтобы держать ИИ под контролем: архитектуру, CLAUDE.md, типизацию, тесты, безопасность. Но за 2025–2026 годы устоялся ещё один слой — *как* именно вести с ИИ рабочий процесс на уровне сессии, задачи и команды. Часть этих подходов — конкретные фичи Claude Code, часть — общеиндустриальные паттерны и инструменты, которые работают независимо от вендора.

### Loop coding — агентные циклы

**Идея:** ИИ не просто отвечает на один запрос, а крутит цикл «написал код → прогнал тесты/линтер → исправил → повторил», пока не выполнится условие остановки — тест зелёный, чек-лист пройден, лимит попыток исчерпан. Человек включается на входе (ставит цель) и на выходе (ревью), а не на каждой итерации.

В [блоге Anthropic по loop engineering](https://claude.com/blog/getting-started-with-loops) цикл прямо определяется как «агент, повторяющий цикл работы, пока не выполнится условие остановки», и выделяются несколько типов циклов в Claude Code:

- **Turn-based** — ручная итерация: вы просите сделать компонент, смотрите результат, просите поправить
- **Goal-based (`/goal`)** — цель со критерием успеха и лимитом попыток, например: «доведи Lighthouse-скор главной страницы до 90+, максимум 5 попыток»
- **Time-based (`/loop`, `/schedule`)** — регулярная работа по расписанию: «каждые 5 минут проверяй мой PR, отвечай на комментарии ревью и чини упавший CI»
- **Проактивные циклы** — событийные автономные рутины для сложной оркестрации

Проактивный цикл стоит развернуть отдельно — это не столько фича Claude Code, сколько общий архитектурный паттерн 2026 года: агент работает непрерывно, не дожидаясь синхронного промпта, общается через мессенджер или почту вместо CLI-сессии в IDE, а действия — бронирование, отправку писем, правки — совершает по событию (входящее сообщение, таймер, cron-задача), а не потому что человек в этот момент что-то печатает. Наглядный пример такой архитектуры — [OpenClaw](https://github.com/openclaw/openclaw), open-source (MIT) личный ассистент, который его разработчики описывают как «личного ИИ-ассистента, который учится и растёт вместе с вами, работая на ваших собственных устройствах». Технически это локальный **Gateway** — единая управляющая плоскость, которая маршрутизирует события между более чем десятком мессенджеров (WhatsApp, Telegram, Slack, Discord, Signal, iMessage и другими) и произвольным LLM-бэкендом (агент model-agnostic: Claude, GPT, Gemini или локальная модель через Ollama), плюс реестр навыков и cron-планировщик, который и превращает разовые команды в постоянно висящие фоновые рутины. [Разбор архитектуры OpenClaw](https://bibek-poudel.medium.com/how-openclaw-works-understanding-ai-agents-through-a-real-architecture-5d59cc7a4764) показывает, что внутри это тот же ReAct-цикл (Reason + Act), что и в loop coding выше: результат каждого вызова инструмента возвращается в контекст и запускает следующую итерацию — разница только в том, что здесь нет человека, который бы разово запускал цикл заново.

Что делает такой всегда-включённый агент управляемым, а не просто рискованным, держится на трёх конкретных механизмах, а не на одних обещаниях. Во-первых, тот же принцип разобран в [Части V](#часть-v-безопасность): необратимые или рискованные действия должны проходить через явный gate подтверждения, а не выполняться автоматически — сам OpenClaw, по [собственному описанию проекта](https://github.com/openclaw/openclaw), запускает второстепенные сессии в изолированной песочнице на Docker, SSH или OpenShell-бэкенде, а не в основном окружении пользователя. Во-вторых, изоляция выполнения кода вынесена в отдельный инфраструктурный слой: [E2B](https://e2b.dev) — сервис песочниц для ИИ-агентов на Firecracker microVM — по [собственной документации](https://e2b.dev) устроен так, что «каждая песочница работает на Firecracker — microVM, созданной для выполнения ненадёжных (untrusted) воркфлоу», что изолирует файловую систему и сеть агента от хоста и от параллельных сессий при времени старта в секунды. В-третьих, стандартизация самого канала между агентом и внешними сервисами — это MCP, разобранный [ниже](#инфраструктурные-фичи-2026): именно он переводит «агент действует в реальном мире» из ad hoc интеграции под каждый сервис в предсказуемый, декларируемый набор инструментов, а не в самописный скрипт с произвольным доступом.

Для контраста полезно увидеть тот же паттерн в соседней нише — автономном кодировании без личного ассистента: [Devin от Cognition](https://devin.ai) выполняет задачу целиком в собственном облачном изолированном окружении (планирует, пишет, тестирует и деплоит код без пошагового участия человека), а открытый [OpenHands](https://github.com/OpenHands/OpenHands) реализует тот же принцип «песочница + автономный цикл» как self-hosted MIT-инструмент, куда можно подключить Claude или любую другую модель по собственному ключу.

Ключевое требование для любого loop — **проверяемый чек**, который ИИ может прогнать сам: тест, сборка, линтер, скриншот. [Официальный гайд по best practices](https://code.claude.com/docs/en/best-practices) формулирует это так: «дайте Claude чек, который он может прогнать — тест, сборку, скриншот для сравнения. Это разница между сессией, за которой вы наблюдаете, и той, от которой можно отойти». Там же описаны три уровня жёсткости цикла: обычный промпт («прогони тест и итерируй»), условие `/goal` (отдельный проверяющий агент перепроверяет после каждого хода) и **Stop hook** — детерминированный скрипт-гейт, который не даёт циклу завершиться, пока проверка не пройдена.

На практике это часто выглядит как **CI-gated loop**: ИИ пушит код, CI гоняет тесты, если красный — ИИ читает лог и чинит сам, без участия человека на этом шаге. [Разбор от Lenny's Newsletter](https://www.lennysnewsletter.com/p/how-i-ai-how-to-write-ai-agent-loops) приводит пример такого цикла — ежедневный проход по открытым PR, который сам чинит упавший CI и подтягивает свежие ревью-комментарии. Автор там же предупреждает о главном риске: расплывчатый критерий успеха заставляет цикл крутиться бесконечно и сжигать токены — критерий должен быть таким же конкретным, как задача для нового сотрудника.

### Skills — переиспользуемые навыки

**Проблема, которую решают Skills:** одна и та же процедура (сгенерировать миграцию, оформить PR по вашим правилам, собрать отчёт по шаблону) объясняется ИИ заново в каждой сессии.

**Skills** — это папки с инструкциями, скриптами и ресурсами, которые ИИ подгружает по мере необходимости, а не держит в контексте постоянно. Anthropic описывает их как «кастомные обучающие материалы, которые позволяют упаковать экспертизу и сделать Claude специалистом именно в том, что важно вам» — фича [была анонсирована 16 октября 2025 года](https://claude.com/blog/skills). Технически skill — это папка с файлом `SKILL.md`: Claude сканирует доступные skills, находит подходящий по описанию и подгружает только нужный минимум файлов, не раздувая контекст. Инженерный разбор архитектуры — в [блоге Anthropic Engineering](https://www.anthropic.com/engineering/equipping-agents-for-the-real-world-with-agent-skills).

```markdown
---
name: api-conventions
description: REST API design conventions for our services
---
# API Conventions
- Use kebab-case for URL paths
- Use camelCase for JSON properties
- Always include pagination for list endpoints
```

Отличие от CLAUDE.md: CLAUDE.md грузится **в каждой** сессии целиком, skill — **только когда релевантен**. Правило простое: то, что нужно всегда, — в CLAUDE.md; то, что нужно иногда, — в skill.

### Sub-agents и мультиагентная оркестрация

**Идея:** вместо одного агента, который последовательно читает весь проект, задача делится между несколькими специализированными агентами, которые работают параллельно и не засоряют друг другу контекст.

В Claude Code есть два разных механизма, и путать их не стоит — [официальная документация](https://code.claude.com/docs/en/agent-teams) чётко их разделяет:

- **Subagents** — лёгкие воркеры со своим контекстным окном, которые получают задачу, выполняют её и отчитываются **только основному агенту**. Хороши для research (`используй subagent, чтобы изучить, как у нас работает refresh токенов`) и для верификации после реализации (`используй subagent, чтобы проверить edge cases в этом коде`)
- **Agent teams** — несколько независимых сессий с общим списком задач, которые обмениваются сообщениями **друг с другом**, а не только с лидером. Подходят для research/review с разных углов, параллельной работы над независимыми модулями, отладки через конкурирующие гипотезы

Типичные паттерны оркестрации:

- **Fan-out/fan-in** — задача дробится на N независимых подзадач (например, миграция 2000 файлов), каждая обрабатывается отдельным вызовом `claude -p`, результаты сводятся воедино
- **Supervisor / lead-agent** — один агент координирует, назначает задачи, синтезирует результаты; воркеры не видят общей картины
- **Writer/Reviewer** — одна сессия пишет реализацию, вторая (в чистом контексте, без «предвзятости» к только что написанному коду) ревьюит диф; в [гайде по best practices](https://code.claude.com/docs/en/best-practices) это отдельно рекомендуемый паттерн — «свежий контекст улучшает ревью, потому что Claude не тяготеет к коду, который сам только что написал»

Минус — токены и координационные издержки растут быстрее, чем польза: официальная документация советует начинать с 3–5 агентов и использовать оркестрацию только там, где параллельность реально ускоряет работу, а не для линейных задач с зависимостями.

Subagents и agent teams в Claude Code — конкретная реализация от одного вендора; сам паттерн мультиагентной оркестрации существует и как самостоятельный тулинг для тех, кто строит собственные агентные системы, а не работает напрямую в Claude Code. Самый распространённый вариант — **[LangGraph](https://docs.langchain.com/oss/python/langgraph/overview)** от LangChain: низкоуровневый фреймворк, где мультиагентный процесс описывается явным графом узлов и рёбер с общим состоянием, которое переносится от шага к шагу, включая циклы — то есть агент может вернуться на предыдущий шаг при новой информации, а не только двигаться по прямой, как в более декларативной оркестрации Claude Code. Стоит знать и о соседях по нише: **[CrewAI](https://github.com/crewAIInc/crewAI)** делает акцент на быстром прототипировании ролевых команд агентов, а **[AG2](https://ag2.ai/)** — открытое продолжение Microsoft AutoGen, который сам перешёл в режим поддержки, — и **[OpenAI Agents SDK](https://openai.github.io/openai-agents-python/)**, пришедший на смену экспериментальному Swarm, со своей моделью хендоффов между агентами.

### Spec-driven программирование

Это отдельная категория тулинга, которую не стоит путать с [Research-Plan-Implement](#research-plan-implement) из Части III. RPI — это письменный план, который вы читаете и одобряете вручную перед кодированием. **Spec-driven development** — более формальный подход: спецификация оформляется в структурированном, машиночитаемом виде и становится контрактом, по которому инструмент сам генерирует план, задачи и код, а на каждом шаге может себя проверить.

Два заметных инструмента, оформившихся в 2025 году:

- **[GitHub Spec Kit](https://github.blog/ai-and-ml/generative-ai/spec-driven-development-with-ai-get-started-with-a-new-open-source-toolkit/)** (анонсирован 2 сентября 2025) — открытый тулкит с четырёхфазным циклом: `Specify` (агент превращает описание в детальную спецификацию с пользовательскими сценариями) → `Plan` (техническая стратегия под ваши ограничения и стек) → `Tasks` (спецификация и план дробятся на маленькие проверяемые задачи) → `Implement` (агент реализует задачи последовательно, вы ревьюите небольшие дифы, а не гигантские дампы кода). Идея, как формулирует GitHub, — сместить источник истины с «код — это истина» на «намерение — это истина»
- **[AWS Kiro](https://kiro.dev/blog/introducing-kiro/)** (анонсирован 14 июля 2025, доступен по подписке с марта 2026) — агентная IDE на базе Code OSS, где по умолчанию нельзя сгенерировать код без прохождения спецификации: запрос вроде «добавь систему отзывов» разворачивается в пользовательские истории с критериями приёмки в нотации **EARS** (Easy Approach to Requirements Syntax), затем в технический дизайн (схемы данных, интерфейсы, эндпоинты) и только потом — в упорядоченные задачи с интеграционными тестами

Спецификация в этом подходе исполняема и проверяема — так же, как тесты в TDD, только на уровне требований, а не кода. Это более тяжеловесный процесс, чем RPI, и оправдан для сложных фич и продакшн-систем, а не для мелких правок.

### Context engineering

Не путать с «[Контекст и Workspace](#контекст-и-workspace)» из Части III — там речь про то, *когда* открывать новую сессию. Здесь — про то, *что именно* модель видит в момент генерации внутри уже открытой сессии.

Термин, который в 2025 году потеснил «prompt engineering»: если prompt engineering — это про формулировку конкретного запроса, то **context engineering** — про то, что модель вообще видит в момент генерации: CLAUDE.md, история диалога, skills, результаты вызова инструментов, содержимое файлов. Сдвиг в терминологии закрепился после того, как [Andrej Karpathy предложил термин «context engineering»](https://neo4j.com/blog/agentic-ai/context-engineering-vs-prompt-engineering/) как более точное описание реальной работы по выстраиванию серьёзных ИИ-приложений.

Для программирования это не абстракция, а конкретная дисциплина: то, что уже описано в Части I (CLAUDE.md) и в этой части (Skills, subagents) — это и есть context engineering на практике. Отдельная ветка того же подхода — **RAG для кода**: индексация кодовой базы, документации и внутренних вики в векторное хранилище, чтобы ИИ подтягивал релевантный контекст по запросу, а не полагался только на то, что попало в промпт. [Обзор RAG для инженерных команд](https://www.kinde.com/learn/ai-for-software-engineering/ai-agents/rag-for-engineers-build-a-retrieval-augmented-coding-assistant-in-20-minutes/) описывает это как способ сделать ассистента «знающим партнёром», глубоко в курсе именно вашей архитектуры и конвенций — на практике в Claude Code эту роль отчасти закрывает Tool Use / MCP (см. [ниже](#инфраструктурные-фичи-2026)), который подключает ИИ напрямую к базам данных, трекерам задач и документации.

**Векторное хранилище** здесь — это база данных, которая хранит эмбеддинги (числовые векторы) кусков кода и документации и находит контекст по семантической, а не текстовой близости. Для RAG по коду на практике используют [Pinecone](https://www.pinecone.io/) (полностью управляемый сервис — «RAG без эксплуатации инфраструктуры БД»), [Weaviate](https://weaviate.io/) (гибрид векторного и keyword-поиска в одном запросе), [Qdrant](https://qdrant.tech/) (написан на Rust, силён в поиске с тяжёлой фильтрацией по метаданным), [Milvus](https://milvus.io/) (распределённый поиск для очень больших датасетов), [Chroma](https://www.trychroma.com/) (самый быстрый старт для прототипов) и [pgvector](https://github.com/pgvector/pgvector) — расширение самого PostgreSQL, добавляющее векторный тип и индексы прямо в реляционную БД, если отдельная векторная СУБД не оправдана; [сравнение вариантов и типичные сценарии их использования — у Braintrust](https://www.braintrust.dev/articles/best-vector-databases-for-rag-2026). У всех один и тот же системный изъян: [InfoQ формулирует его так](https://www.infoq.com/articles/vector-search-hybrid-retrieval-rag/) — «эмбеддинги находят похожее на ваш запрос, а не совпадающее с ним точно», модель «создана находить похожее, а не идентичное». Для кода это значит: семантический поиск подберёт функции с похожей лексикой, но не даёт точного ответа на вопрос «кто вызывает эту функцию» или «что реализует этот интерфейс» — там, где нужна не похожесть, а точная структурная связь.

Для таких запросов существует другая техника — **граф кода**: код парсится (обычно через [tree-sitter](https://github.com/tree-sitter/tree-sitter) или language server / LSP) не в плоские текстовые эмбеддинги, а в граф символов, определений, ссылок, вызовов и типовых отношений. Разницу хорошо формулирует разбор open-source проекта [CodeGraph](https://github.com/colbymchenry/codegraph) — локального MCP-сервера на tree-sitter для Claude Code, Cursor и других агентных редакторов: «семантический поиск не понимает отношений... он выдаёт наводки — ранжированные по схожести подсказки — и ИИ должен затем проверять связи, читая файлы одну за другой», тогда как явный граф вызовов «даёт ИИ ответы — детерминированные структурные факты — вместо наводок» ([разбор на Medium](https://medium.com/kd-agentic/codegraph-the-open-source-knowledge-graph-that-makes-ai-coding-tools-dramatically-cheaper-190f8b89f8a7)). Тот же принцип лежит в основе точной кодовой навигации [Sourcegraph](https://sourcegraph.com/blog/semantic-code-search-what-it-is-and-how-it-works): компания прямо пишет, что «чистый векторный поиск — полезный строительный блок, но не законченное решение», и комбинирует его с SCIP-индексами, которые дают «понимание символов, их определений и кросс-репозиторных ссылок с точностью компилятора» — именно это позволяет инструменту трассировать всех вызывающих и все зависимости функции, а не только файлы, где она упомянута по имени. На практике сильнее всего работает именно комбинация: векторный поиск — для нечёткого семантического «найди что-то похожее», граф кода — для точных структурных вопросов, на которые векторное сходство в принципе не может ответить.

### Vibe coding — противоположный полюс

Полезно назвать и подход, которому вся эта статья противопоставлена. **Vibe coding** — термин [Andrej Karpathy из твита февраля 2025 года](https://x.com/karpathy/status/1886192184808149383): «полностью отдаться вайбам... и забыть, что код вообще существует» — писать код через диалог с ИИ, не читая и не проверяя, что получилось. В 2025 году это слово [стало словом года по версии Collins Dictionary](https://www.coderabbit.ai/blog/a-semantic-history-how-the-term-vibe-coding-went-from-a-tweet-to-prod), что говорит скорее о массовости практики, чем о её надёжности.

Vibe coding — законная техника для одноразовых прототипов и экспериментов, где скорость важнее качества. Но именно из-за него появился спрос на противоположный полюс — spec-driven development и TDD-first подходы: [пост GitHub про Spec Kit](https://github.blog/ai-and-ml/generative-ai/spec-driven-development-with-ai-get-started-with-a-new-open-source-toolkit/) прямо называет vibe coding практикой, которая «отлично подходит для быстрых прототипов, но менее надёжна для серьёзных, критичных для бизнеса приложений». Вся система, описанная в Частях I–VI этой статьи, — это по сути системный отказ от vibe coding в пользу контролируемого процесса.

### TDD-first агентное кодирование

Вариант Research-Plan-Implement, заточенный конкретно под тесты: **сначала пишется падающий тест, только потом — код, который заставляет его пройти.** [Официальный гайд Claude Code по best practices](https://code.claude.com/docs/en/best-practices) прямо советует такую формулировку задачи вместо расплывчатой: не «почини баг логина», а «напиши падающий тест, который воспроизводит проблему, затем почини её» — падающий тест здесь и есть проверяемый критерий для loop coding из начала этой части.

Ключевое правило дисциплины: **тесты пишет и одобряет человек (или отдельная сессия), а не тот же агент, что пишет реализацию** — иначе ИИ может незаметно подогнать тест под свой баг вместо того, чтобы исправить код. На практике это тот же паттерн Writer/Reviewer, что и в оркестрации sub-agents: один агент (или человек) владеет тестом-спецификацией, другой — реализацией, и один не может смошенничать за счёт другого.

```
❌ Дать ИИ карт-бланш: «добавь валидацию email при регистрации»
   → тест и реализация написаны одним проходом, слабый regex
     («есть символ @») проходит мимо обеих сторон незамеченным

✅ TDD-first: сначала тест
   it('rejects an email without a domain', () => {
     expect(isValidEmail('user@')).toBe(false);
   });
   → тест красный → ИИ пишет реализацию → тест зелёный → готово
```

### Human-in-the-loop и парное программирование с ИИ

Даже в самом автоматизированном loop человек остаётся на входе и на выходе. Практика показывает, что удобнее всего расписывать это через роли классического парного программирования: **driver** пишет код, **navigator** думает наперёд и проверяет. В работе с ИИ [эти роли применимы к самому ИИ](https://medium.com/@peter.heller/the-navigator-and-the-driver-a-new-model-for-ai-pair-programming-28ad5b0ab215) — иногда ИИ водит, а вы штурманите и тормозите на поворотах, иногда наоборот. [Обзор IBM по AI pair programming](https://www.ibm.com/think/topics/ai-pair-programming) описывает это как продолжение классического парного программирования, где ИИ берёт на себя роль второго разработчика за той же «клавиатурой».

Продуктивный режим — оставаться навигатором: давать намерение и оценивать результат, не принимать правки не глядя. Это прямое продолжение принципа Части IV (Code review) и Части V (Безопасность): чем длиннее автономный прогон, тем важнее чекпоинт, на котором человек утверждает план или ревьюит диф, прежде чем агент пойдёт дальше. В Claude Code это формализовано через **plan mode** — агент читает код и предлагает план, но не трогает файлы, пока вы не одобрите; [гайд по best practices](https://code.claude.com/docs/en/best-practices) прямо рекомендует этот режим для задач, где неясен подход или задеты несколько файлов, и советует пропускать его только для тривиальных правок вроде опечатки.

### RAG и computer-use агенты

Две смежные практики, которые расширяют, где именно ИИ может действовать:

**RAG для кода** уже упомянут в контексте context engineering выше — это способ подмешать в контекст релевантные куски кодовой базы и документации через поиск, а не полагаться на то, что попало в промпт вручную.

**Computer-use / browser-driving агенты** — ИИ управляет экраном или браузером напрямую: делает скриншоты, кликает, заполняет формы, — вместо заранее написанных интеграций под конкретный сервис. Anthropic описывала это как возможность «управлять любым софтом визуально, используя скриншоты в качестве глаз» ещё в [анонсе Computer Use для Claude 3.5](https://www.anthropic.com/news/3-5-models-and-computer-use) (2024 год, для сравнения с историческим контекстом). К 2026 году эта возможность оформилась в [интеграцию Claude Code с расширением Claude for Chrome](https://code.claude.com/docs/en/chrome): агент открывает страницу в браузере, читает консольные ошибки, сетевые запросы и состояние DOM напрямую, находит причину и сам предлагает исправление в коде — полезно там, где баг воспроизводится только в браузере, а API у стороннего сервиса нет.

### Инфраструктурные фичи 2026

Подходы выше — про то, как организовать работу с ИИ. Ниже — не подходы, а инфраструктурные возможности, которые их ускоряют и удешевляют, независимо от выбранного подхода:

**Prompt Caching** — кеширование повторяющегося контекста (CLAUDE.md, структура проекта, системные промпты). По [официальной документации по кешированию промптов](https://platform.claude.com/docs/en/build-with-claude/prompt-caching), попадание в кеш (cache hit) стоит **0.1x от цены обычного input-токена — то есть на 90% дешевле обычного ввода**, а запись в кеш дороже базовой цены (1.25x для 5-минутного TTL, 2x для часового). [Страница ценообразования Anthropic](https://platform.claude.com/docs/en/about-claude/pricing#prompt-caching) уточняет порог окупаемости: кеш окупается уже после одного повторного чтения при 5-минутном TTL и после двух чтений при часовом. Первый запрос действительно чуть дороже (это запись кеша), но каждый следующий в пределах TTL — заметно дешевле и с меньшей задержкой до первого токена, особенно на длинных документах.

**Thinking Mode** — длинное мышление (extended thinking) для сложного кода: ИИ тратит больше токенов на рассуждение перед ответом, результат обычно лучше на сложных задачах. Это настраиваемый, не бесплатный режим: [гайд Claude Code по управлению расходами](https://code.claude.com/docs/en/costs#adjust-extended-thinking) отмечает, что thinking-токены оплачиваются как output-токены, а бюджет по умолчанию может достигать десятков тысяч токенов на запрос — для простых задач его стоит снижать через `/effort` или переменную `MAX_THINKING_TOKENS`.

**Tool Use / MCP** — ИИ может вызывать инструменты прямо во время работы через [Model Context Protocol](https://code.claude.com/docs/en/mcp). Запуск тестов, линтеров, проверка архитектуры — всё это может делать ИИ автоматически (о цене этого удобства — в [«Точные промпты вместо широкого сканирования»](#точные-промпты-вместо-широкого-сканирования) из Части VI).

**Batch Processing** — массовая асинхронная обработка однотипных задач. [Официальная документация по Batch API](https://platform.claude.com/docs/en/build-with-claude/batch-processing) подтверждает: обработка через Message Batches API даёт **скидку 50% на входные и выходные токены** по сравнению с обычными запросами ценой отложенного ответа — большинство батчей завершается быстрее часа, но гарантированный срок ответа — 24 часа.

### Результат: реальная экономия без техдолга

Если вы следуете этой системе:

2. **Research-Plan-Implement** — контроль перед кодированием
3. **Типизация** — strict type checking (TypeScript, mypy, etc.)
4. **Тесты 80%+** — фильтр логических ошибок
5. **Code review** — финальная проверка
6. **Безопасность** — запрещённые команды, review, бэкапы
7. **Скрипты для рутины** — экономия денег на повторных запросах
8. **Современные фичи ИИ** — Skills, Caching, Sub-agents, Thinking

Вы получаете:
- **2-3x прирост производительности** без роста ошибок
- **Техдолг не растёт** — контроль на каждом этапе
- **Экономия денег** — меньше запросов благодаря скриптам и кешированию

### Как начать прямо сейчас

**Неделя 1:**
- Выберите архитектурный подход:
  - **Для большинства:** Feature-based (проще контролировать AI)
  - **Для сложной логики:** Clean Architecture
  - **Для больших корпоративных систем:** DDD (требует опыта)
- Опишите его в CLAUDE.md
- Настройте dependency-cruiser

**Неделя 2:**
- Внедрите Research-Plan-Implement процесс
- Требуйте план перед кодированием

**Неделя 3:**
- Включите strict type checking для вашего языка:
  - **TypeScript:** `"strict": true` в tsconfig.json
  - **Python:** `mypy --strict` в CI
  - **Go:** обязательно проверяйте ошибки
  - **Java/Kotlin:** включите null-safety проверки
- 80%+ покрытие тестами обязательно

**Неделя 4+:**
- Скрипты для рутины
- Фичи ИИ (caching, skills)
- Постоянное совершенствование

### Главный совет

**Начните с малого.** Не нужно всё внедрять сразу. Главное — **дисциплина и контроль**. Всё остальное придёт со временем.

Правильная система даст вам:
- Контроль над кодом (вы знаете где что)
- Скорость (ИИ ускоряет, не создавая проблем)
- Чистоту (техдолг не растёт)
- Уверенность (тесты и типизация подстраховывают)

Это не просто советы. Это система, которая работает в реальных проектах.
